Java Inner classes

2017-07-31 programming, inDepth

There are four main types of nested classes in Java; Inner, Local, Anonymous and Static which fall mainly into two groups, member level and local level.

The inner class and static both apply to nested classes made at the member level (fields and methods).

Local and Anonymous classes are defined within methods.

Inner Classes

Briefly an inner class can:

  1. Be declared public, private, or protected or use default access
  2. Extend any class and implement interfaces
  3. Be abstract or final
  4. Not declare static fields or methods
  5. Access members of the outer class including private members.

Because a member inner class is not static we first have to instantiate the outer class before we can try instantiate and access the inner class.

public class Outer {
  private String name = "Bob";

  public class Inner {
    public void go(){
        System.out.println(name);
        }
  }

  public static void main(String[] args) {
    Outer outer = new Outer();
    Inner inner = outer.new Inner();
    inner.go();
  }
}

If we compile the above code

javac Outer.java

We get two files

Outer.class and Outer$Inner.class

An interesting thing to note is point number 5 in our list, the inner class can access private members of the outer class, naturally we should be able to do so, methods can access class variables as per the normal rules for scoping. However we have a nested class and private members are only known to that class.

So how does Java reconcile the two, that is; no class should be able to naturally access another classes private members (insert memes here) but still allow for ‘normal’ block scoping?

In short, composition.

Below we can see the decompiled version of our Outer class file

javap -p Outer
Compiled from "Outer.java"
public class Outer {
  private java.lang.String name;
  public Outer();
  public static void main(java.lang.String[]);
  static java.lang.String access$000(Outer);
}

Here we can see our usual stuff, our String field, a default constructor added by Java, most interesting is the last method which was added by Java access$000.

It is a default-package level static method that returns a string and takes an instance of the type Outer.

Looking now to our Inner class, the decompiled version looks like this.

javap -p Outer\$Inner
Compiled from "Outer.java"
public class Outer$Inner {
  final Outer this$0;
  public Outer$Inner(Outer);
  public void go();
}

We see java has added a constructor that takes an Outer object and assigns it to a final field. Since this is a member inner class Java will automatically pass an instance of Outer to the Inner class.

Hopefully we can sort of see how this works.

  1. Outer class provides a method to access a type from itself.
  2. Inner class receives an instance of the outer type.
  3. Inner class calls the access method on the outer class passing in its instance of the outer class.

We can confirm this by verbosely decompiling our inner class.

javap -verbose Outer\$Inner

The decompiled access method simply returns the value requested from the instance passed. Since the object being passed is the same type as the class it is allowed to access private members like normal, we can confirm this by looking at the decompilation.

The synthetic flag simply means Java has generated this method and not the developer.

In fact since there is no magic, just composition and what some could argue is a technicality (a class can read its own private member even if its from another reference of the same type and not this) we could re-write our original to mimic this behaviour.

Our re-wrttien example would look like this

public class Outer {
  private String name = "Bob";

// ------------ start inner class -----------
  public class Inner {
      final Outer outer;
      public Inner(Outer outer) {
          this.outer = outer;
      }

      public void go(){
        System.out.println(Outer.access$000(this.outer));
        }
  }
// ------------ end inner class --------------

      static String access$000(Outer outer) {
        return outer.name; // since we are in the Outer class we can access private fields even if the instance  passed is not `this`
    }

  public static void main(String[] args) {
    Outer outer = new Outer();
    Inner inner = outer.new Inner(outer);
    inner.go();
  }
}

Interesting to note is that the code that is generated by Java is almost package level meaning other classes in the declared packaged could access these generated methods or field.

The access method is static and as such is not subjected to virtual invocation, essentially meaning it can’t be overridden by a sub class, as static method can only be shadowed and not overridden.

Static Nested Classes

Similar to Inner classes but the nested class is tied to the outer class an not an instance, this means unlike Member Inner classes a static nested class does not require an instance of the outer class to be present.

Briefly:

  1. The nesting creates a namespace because the enclosing class name must be used to refer to it.
  2. It can be made private or use one of the other access modifiers to encapsulate it.
  3. The enclosing class can refer to the fields and methods of the static nested class.

The static nested class works much the same as an instance inner class, but with the usual static semantics (you cant call instance members from a static context), static can only access static members, to access instance methods you have to create the object.

A static example of the Nested class in the previous topic could look like this.

 public class Enclosing {
     private static String name = "Bob";

     static class Nested {
         private void go() {
             System.out.println(name);
        }
    }

    public static void main(String[] args) {
        Nested nested = new Nested();
        nested.go();
        } 
}

It is important to remember that even though it is a nested static class, it is a class and can still be instantiated. Here we can create an instance of the static class from the main method as the main method itself is static, hence we are referencing a static member from a static context which is perfectly fine.

Instance methods can access static variables just like normal even if it’s in the outer class like here.

Java uses the same idea of composition for classes to get access to other classes private fields.

The decompiled class are shown below

Enclosing.class

▶ javap -p Enclosing
Compiled from "Enclosing.java"
public class Enclosing {
  private static java.lang.String name;
  public Enclosing();
  public static void main(java.lang.String[]);
  static java.lang.String access$000();
  static {};
}

Enclosing$Nested.class

▶ javap -p Enclosing\$Nested
Compiled from "Enclosing.java"
class Enclosing$Nested {
  Enclosing$Nested();
  private void go();
  static void access$100(Enclosing$Nested);
}

We can see because the inner class is static it does not require the outer class to be instantiated to come into scope, hence we never pass in a instance of the outer class to the inner classes constructor like an instance inner class, instead each class provides its own static methods to access each others data, in this case the static inner class generates static void access$100(Enclosed$Nested) to access the go method and the outer class provides the static java.lang.String access$000(); to access the name field. This latter method is conceptually the same as a getter but in a static way.

Once again this static getter is interesting as it has package/default as its access modifier meaning other classes in the same package can conceptually retrieve the value even though the developer obviously never had this intention.

A key primary difference between the static and non static nested class is that the nested static class generates static wrappers for its instance methods, here Enclosing$Nested creates static void access$100(Enclosing$Nested); which delegates a call to the go method.

However unlike the instance inner class Java has to use a special invocation internally.

javap -p -verbose Enclosing$Nested

The jvm specs define invoke special as

The difference between the invokespecial instruction and the invokevirtual instruction (§invokevirtual) is that invokevirtual invokes a method based on the class of the object. The invokespecial instruction is used to invoke instance initialization methods (§2.9) as well as private methods and methods of a superclass of the current class.

A re-written example may look like


 public class Enclosing {
     private static String name = "Bob";

// ------- Start static inner class -----------
     static class Nested {
         private void go() {
             System.out.println(Enclosing.access$000());
        }
        
        static void access$100(Nested nested) {
         nested.go();
        }
    }
// ------- End static inner class -----------

   static String access$000() {
       return name;
   }

    public static void main(String[] args) {
        Nested nested = new Nested();
        Nested.access$100(nested); // essentially  --> nested.go();
        } 
}

Hopefully we can see that inner classes and static inner classes are roughly the same and static classes still hold similar semantics to those of static fields vs instance fields and static methods vs instance methods.

Next we’ll look at local inner classes and anonymous inner classes and how they are essentially the same thing.

Local Inner classes

A local inner class is a nested class defined within a method. Like local variables, a local inner class declaration does not exist until the method is invoked, and it goes out of scope when the method returns.

Local classes are slightly weird in that they inherit their context from the method, that is the get access modifier and whether or not they are static from method they are declared. If the method is static then the local inner class can only access static members unless an instance is explicitly passed to it, however if the enclosing method is not-static than it acts a s regular instance member class.

Having a class inside a method provides some unique differences than member level nested classes (instance or static) most notable will be how Java passes around private data and scoping.

With member inner classes since the inner class is alive as long as the outer class is alive, there is no real concern the outer class could go out of scope while there was an instance of the inner class. As we’ve seen because of this Java passes and instance of the outer class to the inner class to have access to its fields and methods.

With static member classes as long as the class is loaded you can access all static members, so there is no issue there as well.

With local classes however it is not possible to pass in an instance of the outer class to the local class to obtain local method variables, block scoping doesn’t naturally work this way (note an instance of the outer class may be passed to the inner class so it can access member level fields like those of member inner classes)

So how does Java allow the local class to access variables from the same local method ???

In short Java makes a copy of each local variable and passes it to the constructor of the local class. primarily as Java doesn’t really support closures or capturing as it is known in other languages.

What if the local variable changes, causing in-ies between the local variable and the variable in the inner local class? Java does not update the copied over variable passed to the local class.

To stop this situation Java only allows local variables that are used by the local class to be final, that is, there is no way the value can be changed once it is set.

An example might be

public class Outer {

    public void foo() {
         String name = "Bob"; // Java  8+
// -----------start local class----------
        class Inner {
            public void go() {
                System. out .println(name);
                }
            }
// -----------End local class----------
            Inner inner = new Inner();
            inner.go();
        }
        
    public static void main(String[] args) {
        Outer outer = new Outer();
        outer.foo();
    }
}

The decompiled inner class might look like this

  final Outer this$0;
  Outer$1Inner();

}

You’ll notice in our source code we didn’t explicitly put the final keyword although the name variable never changes, Java makes a final copy of it. This is known as an effectively final variable, that is a variable that fits the behaviour of a final variable (it never has its value changed) but does no require an explicit declaration of the final keyword.

Anonymous Inner classes

Quite literally the same as a local inner class, but you do not declare a new class type, instead you often use anonymous classes to instantiate an abstract class or interface on the fly.

As instantiating a class is a statement, it is one of the few places in Java you will see a semi-colon placed after a closing bracket (…};).

Java 8 lambdas are quintessentially anonymous classes, the examples below are mostly equivalent, I say mostly as Java lambdas have properties such as lazy evaluation and the likes, but the source code is equivalent.

Predicate<String> wordFilter = s -> s.length() == 5;

and

Predicate wordFilter = new Predicate<String>() {
    @Override
    public boolean test(String s) {
        return s.length() == 5;
    };

Since lambdas essentially anonymous classes which are the same as local classes, any local variables still have to be final or effectively final, for the lambda to access them.

Hopefully it should be obvious, but with anonymous classes, since you are declaring a class you can’t extend other classes/interfaces or implement interfaces from within the anonymous class.
