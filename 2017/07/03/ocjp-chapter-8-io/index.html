Chapter 8: IO

Chapter goals:

Java I/O Fundamentals

  • Read and write data from the console
  • Use BufferedReader, BufferedWriter, File, FileReader, FileWriter, FileInputStream, FileOutputStream,ObjectOutputStream, ObjectInputStream, and PrintWriter in the java.io.package.

Understanding Files and Directories

Java’s java.io.File is a major component in understanding Java’s IO structure for files and directories.

Conceptualizing the File System

Java describes the following:

A file is record within a file system that stores user and system data.

A directory is a record within a file system that contains files as well as other directories. For simplicity, we often refer to a directory reference as a file record throughout this chapter

A path is a String representation of a fi le or directory within a fi le system.

Many different operating systems use their own file system to store their data, Java attempts to provide a uniform interface for dealing with files and directories across Linux,Mac and Windows.

Introducing the File Class

The File class is used to represent a file or a file record (directory) on a file-system, an instance of the file class is used to represent a path on the file system.

The File can not directly read and write from a from a file, but it can be passed to a number of operations which do read and write from a file.

Creating a File Object

You create can create a File instance by passing a an absolute or relative path (as a string) to the classes constructor. We can also construct a path by passing in a parent file object and a path, to build an absolute path.

File file = new File("/home/smith/data/zoo.txt");
File file = new File("data/zoo.txt"); //assuming we are working in the directory /home/smith/

File parent = new File("/home/smith");
File child = new File(parent,"data/zoo.txt");

As file systems use different path separators, some use backslash and some use forward slash, Java allows us to retrieve the separator via two ways.

System.out.println(System.getProperty("file.separator"));
System.out.println(java.io.File.separator);

When you create a file object, Java does not check for the validity of the file or directory.

File file = new File("/dskfdknfdnf/eworieor/dvmdkl/g/h/fh/data/ooz.xtx");// no issue here even if this is NOT a valid path, Java creates the File object like normal.

This is because a file object represents a path on the file system, only when Java interacts or performs an operation on the file system does it realise, this path does not exist.

Working with a File Object

There a few instance methods you can use when dealing with a file object.

import java.io.File;

public class ReadFileInformation {
  public static void main(String[] args) {
  File file = new File("C:\\data\\zoo.txt");
  System.out.println("File Exists: "+file.exists());

  if(file.exists()) {
    System.out.println("Absolute Path: "+file.getAbsolutePath());
    System.out.println("Is Directory: "+file.isDirectory());
    System.out.println("Parent Path: "+file.getParent());

    if(file.isFile()) {
      System.out.println("File size: "+file.length());
      System.out.println("File LastModified: "+file.lastModified());

    } else {
      for(File subfile: file.listFiles()) {
        System.out.println("\t"+subfile.getName());
      }
    }
  }
  }
}

Here if we pass a valid file we can get

File Exists: true
Absolute Path: C:\data\zoo.txt
Parent Path: C:\data
Is Directory: false
File size: 12382
File LastModified: 1420070400000

But if we pass a directory

File Exists: true
Absolute Path: C:\data
Parent Path: C:\
Is Directory: true
employees.txt
zoo.txt
zoo-backup.txt

Showing how using File to represent both files and directories can cut down on code size.

Introducing Streams

Streams in the java.io.* package are used to often represent a stream of bytes in IO operations, such as reading/writing from a file, or network, they are not related to the new Stream<T> utility API in Java 8.

Stream Fundamentals

Data from a file or network can be represented a stream or flow data, this allows us to work with large files in a reasonable manner, with having to load the whole object into memory (although we could if we wanted to).

In Java all low-level streams work in bytes when dealing with resources, there are higher level streams that provide basic abstraction such as working with characters or blocks of bytes.

Java uses streams whenever it can be beneficial to read and write data from a sequential data source, including files, network connections or the terminal, via System.in, System.err and System.out.

Stream Nomenclature

Byte Streams vs. Character Streams

Differences between Streams and Readers/Writers

  1. The stream classes are used for inputting and outputting all types of binary or byte data.
  2. The reader and writer classes are used for inputting and outputting only character and String data.

Readers adn Writers are often used by the developer as they allow to use a stream without having to worry about any binary or encoding that is happening.

Input and Output

Most classes have to variations depending on which way the data is flowing, in or out. This often relates to low level streams such as InputStream and OutputStream. Readers are often higher level classes that deal with reading in data and writers, output data.

Low-Level vs. High-Level Streams

Generally speaking a low level stream directly interacts with the data source, such as a file, memory or network connection, they also often deal with bytes and do so one at a time, such as the FileInputStream class which reads bytes from a file.

High level streams often wrap a low level stream and provide a more developer friendly experience for common operations, such as reading a line from a file with the default coding already specified; these include FileWriter and BufferedWriter classes.

try (BufferedReader bufferedReader = new BufferedReader(new FileReader("zoo-data.txt"))) {
System.out.println(bufferedReader.readLine());
}

Note high level stream can take and wrap other high level streams as well

try (ObjectInputStream objectStream = new ObjectInputStream( 
    new BufferedInputStream(
        new FileInputStream("zoo-data.txt")))) {
System.out.println(objectStream.readObject());
}

The only low level stream required for the exam are low level file reading stream, FileInputStream, FileOutputStream, FileReader, FileWriter

Stream Base Classes

There are four base abstract classes often used in the java.io package

  1. InputStream
  2. OutputStream
  3. Reader
  4. Writer

Many higher level stream take a reference to the base classes so any calling code can pass any other low level or high level streams or writers.

However input streams can wrap only other input streams and output streams can only wrap other output streams, this is same for readers and writers as well, in many cases mixing these up will cause the compiler to fail.

All examples below fail to compile.

new BufferedInputStream(new FileReader("zoo-data.txt")); // BufferedInputSTREAM can not take a reader
new BufferedWriter(new FileOutputStream("zoo-data.txt")); // BufferedWRITER can not take a stream
new ObjectInputStream(new FileOutputStream("zoo-data.txt")); // We can not mix an input stream with an output stream
new BufferedInputStream(new InputStream()); // InputStream is abstract and hence can not be instantiated.

Decoding Java I/O Class Names

  1. A class with the word InputStream or OutputStream in its name is used for reading or writing binary data, respectively.
  2. A class with the word Reader or Writer in its name is used for reading or writing character or string data, respectively.
  3. Most, but not all, input classes have a corresponding output class.
  4. A low-level stream connects directly with the source of the data.
  5. A high-level stream is built on top of another stream using wrapping.
  6. A class with Buffered in its name reads or writes data in groups of bytes or characters and often improves performance in sequential file systems.
  7. When wrapping a stream you can mix and match only types that inherit from the same abstract parent stream.

Common Stream Operations

Closing the Stream

Since streams are resources they often need to be closed in order to prevent other services from hanging,or un-necessarily consuming the resource, it can also lead to memory and other forms of leaks if left un-closed.

Flushing the Stream

Since writing to a file or network connection is a relatively expensive operation (in terms of time) often, the underlying OS will save data into batches to reduce the number of times data has to be written to the resource. In Java we can force data to be written using the flush() method. Note however using the flush() to liberally can cause a degradation in performance due to consistently hitting the disk or network.

Most implementations of the close() method automatically call the flush method before closing the resource.

Marking the Stream

You can call the mark(int) and reset() on some streams to move to a certain point in the stream. Note only some streams have this functionality and you can check for it via the markSupported() method which returns true or false.

You can specify a limit to read ahead in the mark method and then read upto that many number of bytes, if you want to go back to your original place in the stream you can call reset()

InputStream is = . . . //ABCD
System.out.print ((char)is.read());
if(is.markSupported()) {
  is.mark(100);
  System.out.print((char)is.read());
  System.out.print((char)is.read());
  is.reset();
}

System.out.print((char)is.read());
System.out.print((char)is.read());
System.out.print((char)is.read());

// prints: ABCBCD

If you pass your mark limit and then call reset an exception may be thrown at runtime.

Skipping over Data

The skip(long) method skips over the number of bytes specified and is associated with InputStream’s and Reader’s. The skip method returns the number of bytes actually skipped or a negative number if it reached the end of the file.

InputStream is = . . .// TIGERS
System.out.print((char)is.read()); //prints T
is.skip(2) // skips I,G
is.read();// reads 'E" but does nothing with it
System.out.print((char)is.read());// prints R
System.out.print((char)is.read());// prints S

//prints: TRS
