Understanding Files and Directories

Java’s java.io.File is a major component in understanding Java’s IO structure for files and directories.

Conceptualizing the File System

Java describes the following:

A file is record within a file system that stores user and system data.

A directory is a record within a file system that contains files as well as other directories. For simplicity, we often refer to a directory reference as a file record throughout this chapter

A path is a String representation of a fi le or directory within a fi le system.

Many different operating systems use their own file system to store their data, Java attempts to provide a uniform interface for dealing with files and directories across Linux,Mac and Windows.

Introducing the File Class

The File class is used to represent a file or a file record (directory) on a file-system, an instance of the file class is used to represent a path on the file system.

The File can not directly read and write from a from a file, but it can be passed to a number of operations which do read and write from a file.

Creating a File Object

You create can create a File instance by passing a an absolute or relative path (as a string) to the classes constructor. We can also construct a path by passing in a parent file object and a path, to build an absolute path.

File file = new File("/home/smith/data/zoo.txt");
File file = new File("data/zoo.txt"); //assuming we are working in the directory /home/smith/

File parent = new File("/home/smith");
File child = new File(parent,"data/zoo.txt");

As file systems use different path separators, some use backslash and some use forward slash, Java allows us to retrieve the separator via two ways.

System.out.println(System.getProperty("file.separator"));
System.out.println(java.io.File.separator);

When you create a file object, Java does not check for the validity of the file or directory.

File file = new File("/dskfdknfdnf/eworieor/dvmdkl/g/h/fh/data/ooz.xtx");// no issue here even if this is NOT a valid path, Java creates the File object like normal.

This is because a file object represents a path on the file system, only when Java interacts or performs an operation on the file system does it realise, this path does not exist.

Working with a File Object

There a few instance methods you can use when dealing with a file object.

import java.io.File;

public class ReadFileInformation {
  public static void main(String[] args) {
  File file = new File("C:\\data\\zoo.txt");
  System.out.println("File Exists: "+file.exists());

  if(file.exists()) {
    System.out.println("Absolute Path: "+file.getAbsolutePath());
    System.out.println("Is Directory: "+file.isDirectory());
    System.out.println("Parent Path: "+file.getParent());

    if(file.isFile()) {
      System.out.println("File size: "+file.length());
      System.out.println("File LastModified: "+file.lastModified());

    } else {
      for(File subfile: file.listFiles()) {
        System.out.println("\t"+subfile.getName());
      }
    }
  }
  }
}

Here if we pass a valid file we can get

File Exists: true
Absolute Path: C:\data\zoo.txt
Parent Path: C:\data
Is Directory: false
File size: 12382
File LastModified: 1420070400000

But if we pass a directory

File Exists: true
Absolute Path: C:\data
Parent Path: C:\
Is Directory: true
employees.txt
zoo.txt
zoo-backup.txt

Showing how using File to represent both files and directories can cut down on code size.

Introducing Streams

Streams in the java.io.* package are used to often represent a stream of bytes in IO operations, such as reading/writing from a file, or network, they are not related to the new Stream<T> utility API in Java 8.

Stream Fundamentals

Data from a file or network can be represented a stream or flow data, this allows us to work with large files in a reasonable manner, with having to load the whole object into memory (although we could if we wanted to).

In Java all low-level streams work in bytes when dealing with resources, there are higher level streams that provide basic abstraction such as working with characters or blocks of bytes.

Java uses streams whenever it can be beneficial to read and write data from a sequential data source, including files, network connections or the terminal, via System.in, System.err and System.out.

Stream Nomenclature

Byte Streams vs. Character Streams

Differences between Streams and Readers/Writers

  1. The stream classes are used for inputting and outputting all types of binary or byte data.
  2. The reader and writer classes are used for inputting and outputting only character and String data.

Readers adn Writers are often used by the developer as they allow to use a stream without having to worry about any binary or encoding that is happening.

Input and Output

Most classes have to variations depending on which way the data is flowing, in or out. This often relates to low level streams such as InputStream and OutputStream. Readers are often higher level classes that deal with reading in data and writers, output data.

Low-Level vs. High-Level Streams

Generally speaking a low level stream directly interacts with the data source, such as a file, memory or network connection, they also often deal with bytes and do so one at a time, such as the FileInputStream class which reads bytes from a file.

High level streams often wrap a low level stream and provide a more developer friendly experience for common operations, such as reading a line from a file with the default coding already specified; these include FileWriter and BufferedWriter classes.

try (BufferedReader bufferedReader = new BufferedReader(new FileReader("zoo-data.txt"))) {
System.out.println(bufferedReader.readLine());
}

Note high level stream can take and wrap other high level streams as well

try (ObjectInputStream objectStream = new ObjectInputStream( 
    new BufferedInputStream(
        new FileInputStream("zoo-data.txt")))) {
System.out.println(objectStream.readObject());
}

The only low level stream required for the exam are low level file reading stream, FileInputStream, FileOutputStream, FileReader, FileWriter

Stream Base Classes

There are four base abstract classes often used in the java.io package

  1. InputStream
  2. OutputStream
  3. Reader
  4. Writer

Many higher level stream take a reference to the base classes so any calling code can pass any other low level or high level streams or writers.

However input streams can wrap only other input streams and output streams can only wrap other output streams, this is same for readers and writers as well, in many cases mixing these up will cause the compiler to fail.

All examples below fail to compile.

new BufferedInputStream(new FileReader("zoo-data.txt")); // BufferedInputSTREAM can not take a reader
new BufferedWriter(new FileOutputStream("zoo-data.txt")); // BufferedWRITER can not take a stream
new ObjectInputStream(new FileOutputStream("zoo-data.txt")); // We can not mix an input stream with an output stream
new BufferedInputStream(new InputStream()); // InputStream is abstract and hence can not be instantiated.

Decoding Java I/O Class Names

  1. A class with the word InputStream or OutputStream in its name is used for reading or writing binary data, respectively.
  2. A class with the word Reader or Writer in its name is used for reading or writing character or string data, respectively.
  3. Most, but not all, input classes have a corresponding output class.
  4. A low-level stream connects directly with the source of the data.
  5. A high-level stream is built on top of another stream using wrapping.
  6. A class with Buffered in its name reads or writes data in groups of bytes or characters and often improves performance in sequential file systems.
  7. When wrapping a stream you can mix and match only types that inherit from the same abstract parent stream.

Common Stream Operations

Closing the Stream

Since streams are resources they often need to be closed in order to prevent other services from hanging,or un-necessarily consuming the resource, it can also lead to memory and other forms of leaks if left un-closed.

Flushing the Stream

Since writing to a file or network connection is a relatively expensive operation (in terms of time) often, the underlying OS will save data into batches to reduce the number of times data has to be written to the resource. In Java we can force data to be written using the flush() method. Note however using the flush() to liberally can cause a degradation in performance due to consistently hitting the disk or network.

Most implementations of the close() method automatically call the flush method before closing the resource.

Marking the Stream

You can call the mark(int) and reset() on some streams to move to a certain point in the stream. Note only some streams have this functionality and you can check for it via the markSupported() method which returns true or false.

You can specify a limit to read ahead in the mark method and then read upto that many number of bytes, if you want to go back to your original place in the stream you can call reset()

InputStream is = . . . //ABCD
System.out.print ((char)is.read());
if(is.markSupported()) {
  is.mark(100);
  System.out.print((char)is.read());
  System.out.print((char)is.read());
  is.reset();
}

System.out.print((char)is.read());
System.out.print((char)is.read());
System.out.print((char)is.read());

// prints: ABCBCD

If you pass your mark limit and then call reset an exception may be thrown at runtime.

Skipping over Data

The skip(long) method skips over the number of bytes specified and is associated with InputStream’s and Reader’s. The skip method returns the number of bytes actually skipped or a negative number if it reached the end of the file.

InputStream is = . . .// TIGERS
System.out.print((char)is.read()); //prints T
is.skip(2) // skips I,G
is.read();// reads 'E" but does nothing with it
System.out.print((char)is.read());// prints R
System.out.print((char)is.read());// prints S

//prints: TRS

Working with Streams

The FileInputStream and FileOutputStream Classes

The File*Streams are used to read and write bytes to a stream, they can take a File object in their constructor, denoting which file to read or write from or to.

Most often the read() method which reads a byte is used to retrieve data. When the stream reaches the end of the file it returns -1.

Although the these classes physically deal with bytes, the read() and write() method both return ints so they can use -1 to show the end of the file has been reached.

Both FileInputStream and FileOutputStream methods of read(byte[]) and write(byte[]) can be used to transfer read bytes from a file into a byte array allowing better performance as the program can store parts of the stream in memory and buffer any operations.

import java.io.*;
public class CopyFileSample {
  public static void copy(File source, File destination) throws IOException {
    try (InputStream in = new FileInputStream(source);
      OutputStream out = new FileOutputStream(destination)) {
      int b;
      while((b = in.read()) != -1) {
        out.write(b);
      }
    }
  }
}

Thr above method is how we could copy the contents from one file to another using low level streams.

Note the copy method throws an IOException as the File*Streams both declare FileNotFoundException

The BufferedInputStream and BufferedOutputStream Classes

The BufferedStreams are often used to wrap lower level streams, they provide buffering data operations which in turn can lead to caching, easier seeking and batch read writes,all which help with performance. When using a buffer stream we often pass in a low level stream that we want to operate on and a byte array (our buffer) where the data can be temporarily stored.

import java.io.*;

public class CopyBufferFileSample {
  public static void copy(File source, File destination) throws IOException {
    try (InputStream in = new BufferedInputStream(new FileInputStream(source));
        OutputStream out = new BufferedOutputStream(new FileOutputStream(destination))) {
          byte[] buffer = new byte[1024];
          int lengthRead;
          while ((lengthRead = in.read(buffer)) > 0) {
            out.write(buffer,0,lengthRead); // args = buffer, offset, amount to read in
            out.flush();
          }
    }
  }
}

Here we read in 1kb from the input file, then write the 1kb of data to the output file, finally we call flush at the end of every loop, although a performance downgrade. Calling flush will ensure the buffer is written to disk before receiving the next 1kb of data.

The FileReader and FileWriter Classes

The file reader pair are have a similar role and API to their stream counter parts however the reader deals with chars not bytes and is often, used more when dealing with files. The File reader still have the read(), write(), close() and flush() methods.

The BufferedReader and BufferedWriter Classes

Since a buffered char or char[] is essentially a string, the BufferReader/Writer classes often allow developers to deal with strings instead characters or other primitive types.

public static List<String> readFile(File source) throws IOException {
  List<String> data = new ArrayList<String>();
  try (BufferedReader reader = new BufferedReader(new FileReader(source))) {
    String s;
    while((s = reader.readLine()) != null) {
      data.add(s);
  }
}
return data;
}

Here we check of the buffer reader returns null because the readLine() returns a string (under the hood a char array gets converted into a string)

Comparing the Two Copy Applications

While streams can give us more control when it comes to reading and writing, such as the buffer size the which character set to be used for encoding/decoding, the reader and writer classes provide a more structured way of dealing with text based files.

The reader and writers automatically deal with character encoding (by default the assume UTF-8)

They also provide methods to work with strings instead of primitives or array of primitives making more developer friendly.

However the reader and writer are really only used for text based data, other data such as from networks, or memory will most likely have to accessed via a low level stream.

The ObjectInputStream and ObjectOutputStream Classes

The Object streams allow us to read and write Java objects to and from a data source such as disk.

The process of converting an in-memory object to a stored data format is referred to as serialization

The process of converting stored data format into an in-memory object is referred to as de-serialization

The Serializable Interface

Any classes that want to be serialised must implement the Serializable marker interface, meaning the interface does not declare any methods or variables. Implementing the Serializable denotes that not only is this object serializable but also any instance variables are also serializable.

Attempting to serialise an object which itself or any of its instance variable does NOT implement the serializable interface will throw a NotSerializableException.

If you do want a certain field to be serialized then you can add the transient keyword to the field, this means the field and its value will not be part of the serialization/de-serialization process.

Static variable are not serialized either and would have to be copied to an instance variable if they have to be transferred.

import java.io.Serializable;

public class Animal implements Serializable {
  private static final long serialVersionUID = 1L;
  private String name;
  private int age;
  private char type;
  public Animal(String name, int age, char type) {
    this.name = name;
    this.age = age;
    this.type = type;
  }

  public String getName() { return name; }
  public int getAge() { return age; }
  public char getType() { return type; }
  public String toString() {
    return "Animal [name=" + name + ", age=" + age + ", type=" + type + "]";
  }
}

The above class is completely serializable as all primitives are serializable and the String itself implements the Serializable interface.

Here we also implement a serialVersionUID which must have the type of long and is declared as being static and final. The UID is used by Java to check whether a serialized object can be de-serialized back into an object, as fields and methods may change over time.

Note: The serialVersionUID is static and hence it is not copied over during serialization however, it is a special file descriptor used by the ObjectInput/OutputStream classes, if the developer does not explicitly set the value of this field Java will do so for you, however it is considered best practice for the developer to explicitly state the version, so other developers can see the version, also the default implementation sued by Java can lead to inconsistencies across different JVM and runtime environments. See here for more.

Serializing and Deserializing Objects

Java provides two main classes for serializing and deserializing an object:

  1. ObjectOutputStream - used to convert an in-memory object to a serialized form that can be saved to disk or sent over a network.

  2. ObjectInputStream - used to convert a serialized object from disk to an object that can be read into memory.

The main method to achieve serializing an object using the ObjectOutputStream is the writeObject(Object)

The main method to achieve serializing an object using the ObjectInputStream is the readObject() which returns a java.lang.Object and hence must be cast at runtime.

import java.io.*;
import java.util.*;

public class ObjectStreamSample {
  public static List<Animal> getAnimals(File dataFile) throws IOException, ClassNotFoundException {
  List<Animal> animals = new ArrayList<Animal>();
  try (ObjectInputStream in = new ObjectInputStream(
      new BufferedInputStream(new FileInputStream(dataFile)))) {
    while(true) {
      Object object = in.readObject();
      if(object instanceof Animal)
      animals.add((Animal)object);
    }

  } catch (EOFException e) {
    // File end reached
}
  return animals;
}

public static void createAnimalsFile(List<Animal> animals, File dataFile) throws IOException {
  try (ObjectOutputStream out = new ObjectOutputStream(
      new BufferedOutputStream(new FileOutputStream(dataFile)))) {
    for(Animal animal: animals)
      out.writeObject(animal);
  }
}

public static void main(String[] args) throws IOException, ClassNotFoundException {
  List<Animal> animals = new ArrayList<Animal>();
  animals.add(new Animal("Tommy Tiger",5,'T'));
  animals.add(new Animal("Peter Penguin",8,'P'));

  File dataFile = new File("animal.data");

  createAnimalsFile(animals,dataFile);
  System.out.println(getAnimals(dataFile));
  }
}

//prints: [Animal [name=Tommy Tiger, age=5, type=T], Animal [name=Peter Penguin, age=8,type=P]]

Here we wrap our streams in a BufferedInput/OutputStream for performance reasons.

Also to unlike other streams where we check to see f we’ve reached the end of the file, here catch an EOFException, this is the most common way when dealing with ObjectStreams as we want to make sure we have read every single byte.

As we are casting an object to an instance it is best advised to use the instanceOf operator to ensure we dont get a ClassCastException

We also have handle or re-declare the ClassNotFoundException as the object’s class we are de-serializing may not be available at runtime to that JRE if the object has come from another program.

Finally the object stream classes do not support the serializing or de-serializing null values hence it is important to either enforce null checks or make sure there is no natural way for a variable to become null.

Understanding Object Creation

When de-serializing an object Java essentially «plugs-in» the values to the object from the serialized object, this means Java skips any constructors, instance initialisers or static initialisers as well.

Technically Java uses the first non-serializable objects constructor to create the object and then proxies in the values allowing to by-pass all the «normal» of instantiating or initialising an object.

public class Animal implements Serializable {
  private static final long serialVersionUID = 2L;
  private transient String name;
  private transient int age = 10;
  private static char type = 'C';
  {this.age = 14;}

  public Animal() {
    this.name = "Unknown";
    this.age = 12;
    this.type = 'Q';
  }

  public Animal(String name, int age, char type) {
    this.name = name;
    this.age = age;
    this.type = type;
}
// Same methods as before
. . .
}
//prints: [Animal [name=null, age=0, type=P], Animal [name=null, age=0, type=P]]

Here our name and age fields are transient,so their values are not maintained during the process.

The values of 10,14 and 12 are lost as Java does not use any constructors or initialiser when de-serializing an object.

As for why the type prints «p»

The sample program displays a value for type, as the variable is shared by all instances of the class and is the last value in our sample program.

The PrintStream and PrintWriter Classes

The PrintStream and PrintWriter classes are high-level stream classes that write formatted representation of Java objects to a text-based output stream.

These classes can take an OutputStream and a Writer for their respective constructors.

Both System.out and System.err are PrintStream objects and have the methods write(), print(), println(), format() and printf().

Since none of the print or write methods declare a checked exception we can use the classes checkError() to see of there where any error.

print()

The print method is a heavily overloaded method that can take all primitives, Strings and Object and prints to it’s directed output.

Under the hood many values are converted to the String class before being printed, either using the objects toString() or using String.valueOf() to create string representation of a primitive.

out.print(5); // PrintWriter method
out.write(String.valueOf(5)); // Writer method

out.print(2.0); // PrintWriter method
out.write(String.valueOf(2.0)); // Writer method

Animal animal = new Animal();
out.print(animal); // PrintWriter method
out.write(animal==null ? "null": animal.toString()); // Writer method

println()

println() is essentially identical to print() however it adds a line separator at the end of each print.

Although we can manually pass in a line feed in the print() it is recommend to use println() as Java will deal with any specific OS dependant line separators.

Such as ‘\n’ for Mac/Linux vs ‘\r\n’ for Windows.

format() and printf()

format() and print() both take in a string, locale and any number arguments and return a formatted string. These work identical to String.format()

printf() is identical to format() but is used to make C developers feel more at home (you use printf() in C to print to console, like Java uses print() and println())

public PrintWriter format(String format, Object args. . .)
public PrintWriter printf(String format, Object args. . .)

Sample PrintWriter Application

import java.io.*;

public class PrintWriterSample {
  public static void main(String[] args) throws IOException {
    File source = new File("zoo.log");
    try (PrintWriter out = new PrintWriter(new BufferedWriter(new FileWriter(source)))) {
      out.print("Today's weather is: ");
      out.println("Sunny");
      out.print("Today's temperature at the zoo is: ");
      out.print(1/3.0);
      out.println('C');
      out.format("It has rained 10.12 inches this year");
      out.println();
      out.printf("It may rain 21.2 more inches this year");
    }
  }
}

// It has rained 10.12 inches this year
// It may rain 21.2 more inches this year

Review of Stream Classes

You should be familiar with the following structure of the java.io stream classes.

Other Stream Classes

Other classes not need for the exam but are useful to know are:

The InputStreamReader and OutputStreamWriter both which convert a stream into a reader or writer.

The DataInputStream and DataOutputStream which are similar to ObjectInput/OutputStream but can only serialize/de-serialize Strings and primitives, they also enforce tight data structures meaning the order of serialization and de-serialization is important.

Interacting with Users

Prior to Java 6 if program wanted to interact with a user through a CLI, they would have had to use System.in and System.out. Java 6 introduced the Console class as a better way to interact through a CLI environment.

The Old Way

System.in returns an InputStream that is often wrapped in a BufferedReader, this is to give us access to the readLine() which reads a buffered input when the user hits the «Enter» key. We also dont close the stream as this would stop the program from reading an further input from the user.

import java.io.*;

public class SystemInSample {
  public static void main(String[] args) throws IOException {
    BufferedReader reader = new BufferedReader(new InputStreamReader(System.in));
    String userInput = reader.readLine();
    System.out.println("You entered the following: "+userInput);
}
}

Here we take an input and then print it straight back out.

The New Way

As dealing directly with streams can be repetitive(boiler-plate coding) and low level, the Console class was introduced to help make interacting with use easier.

The console class is created when the JVM starts up and is a singleton class it can be accessed via the System.console() method which returns the console instance.

If the environment does not support text based interactions then calling System.console() will return null.

import java.io.Console;

public class ConsoleSample {
  public static void main(String[] args) {
    Console console = System.console();
    if(console != null) {
      String userInput = console.readLine();
  console.writer().println ("You entered the following: "+userInput);
    }
  }
}

Here this looks somewhat familiar to the Input/OutputStreams example, and under the hood, the Console class uses these classes, but allows us to do so without having to worry about creating, wrapping and closing those streams.

reader() and writer()

The Console class provides access to an instance of Reader and PrintWriter using the methods reader() and writer(), respectively.

The old way used streams (unless the developer themselves wrapped them in a reader or writer), here we use readers and writers so we don’t have to worry about character encoding/decoding.

format() and printf()

As with writers we can use format() and print() to print a formatted string to the user

Console console = System.console();
console.writer().format(new Locale("fr", "CA"),"Hello World");

In the below we can use format() and printf() directly or can use the underlying writer instance

import java.io.*;

public class ConsoleSamplePrint {
  public static void main(String[] args) throws NumberFormatException,IOException {
    Console console = System.console();
    if(console == null) {
      throw new RuntimeException("Console not available");
    } else {
      console.writer().println("Welcome to Our Zoo!");
      console.format("Our zoo has 391 animals and employs 25 people.");
      console.writer().println();
      console.printf("The zoo spans 128.91 acres.");
    }
  }
}

flush()

The flush() method forces any buffered output to be written immediately.

It is recommend to call flush before reading in user input via readLine() or readPassword() so as to ensure you are working with an empty buffer and still dont have any residual characters left.

readLine()

The basic readLine() method retrieves a single line of text from the user, and the user presses the Enter key to terminate it.

import java.io.*;

public class ConsoleReadInputSample {
public static void main(String[] args) throws NumberFormatException, IOException {
  Console console = System.console();
  if(console == null) {
    throw new RuntimeException("Console not available");
  } else {      
    console.writer().print("How excited are you about your trip today? ");
    console.flush();

    String excitementAnswer = console.readLine();
    String name = console.readLine(“Please enter your name: “); //fig1
    Integer age = null;
    console.writer().print("What is your age? ");
    console.flush();

    BufferedReader reader = new BufferedReader(console.reader()); //fig 2
    String value = reader.readLine();
    age = Integer.valueOf(value);

    console.writer().println();
    console.format("Your name is "+name);
    console.writer().println();
    console.format("Your age is "+age);
    console.printf("Your excitement level is: "+excitementAnswer);
    }
  }
}

In the above you can see we have the option to directly call the readLine() method from the console, or we can access the underlying BufferedReader() and then call readLine().

Note all user input is read in as a String and must be converted to its appropriate type, as this is a text based interaction, will almost exclusively be working with strings and primitives.

readPassword()

readPassword() is nearly the same as readLine() except it doesn’t echo back to the user’s screen, this is to prevent people from reading the password over the users shoulder.

The only other notable difference is that readPassword() returns a char[] instead of String. This is for security reasons, as String can be stored in the String pool and read by other parts of the program, until it is garbage collected.

With a character array once you have read the password, you can then manually overwrite all the values essentially «wiping» the password clean from the array.

import java.io.*;
import java.util.Arrays;

public class PasswordCompareSample {
  public static void main(String[] args) throws NumberFormatException,IOException {
    Console console = System.console();
    if(console == null) {
      throw new RuntimeException("Console not available");
    } else {

      char[] password = console.readPassword("Enter your password: ");
      console.format("Enter your password again: ");
      console.flush();
      char[] verify = console.readPassword();
      boolean match = Arrays.equals(password,verify);

      // Immediately clear passwords from memory
      for(int i=0; i<password.length; i++) {
        password[i]='x';
      }

      for(int i=0; i<verify.length; i++) {
        verify[i]='x';
      }

      console.format("Your password was "+(match ? "correct": "incorrect"));
      }
  }
}

We could also have used Array.fill(password,'x') to put dummy data into the arrays.
